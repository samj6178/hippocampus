# UroClub — мини-мессенджер: спецификация и смета

## Цель
Собственное приложение UroClub, независимое от Telegram, с функционалом, похожим на чат с топиками (Topics).

---

## Функциональные требования

### 1. Базовый функционал (MVP)

| Функция | Описание | Сложность |
|---------|----------|-----------|
| **Регистрация/авторизация** | Email + пароль, опционально OAuth (Google/Яндекс) | Средняя |
| **Профиль пользователя** | Имя, аватар, специализация (для врачей) | Низкая |
| **Топики (каналы)** | Список топиков как в Telegram — иконка, название, последнее сообщение, время | Средняя |
| **Закреплённые топики** | 2–3 топика сверху (Уведомления, Совет, События) | Низкая |
| **Чат внутри топика** | Сообщения в реальном времени (WebSocket) | Высокая |
| **Текстовые сообщения** | Отправка/получение, история | Средняя |
| **Вложения** | Фото, файлы (PDF, документы) — как в «Библиотеке» | Средняя |
| **Поиск** | По сообщениям и топикам | Средняя |
| **Уведомления** | Push при новых сообщениях | Средняя |

### 2. Топики по аналогии с UroClub в Telegram

- **Уведомления** (закреплён)
- **Совет** (закреплён)
- **События 2026** (закреплён)
- **Общие вопросы**
- **Отдам в хорошие руки**
- **Библиотека** (файлы, статьи)
- **Оборудование и расходка**
- **Учёба, конференции**
- **Видео операций**
- **Наука** (статьи)
- **Юридическая поддержка**
- **Вакансии**
- **@UroclubGPTbot** — интеграция AI-бота
- **3 дня с экспертом** (спец. события)

### 3. Роли и права

| Роль | Права |
|------|-------|
| Админ | Создание топиков, закрепление, модерация, приглашение |
| Модератор | Удаление сообщений, бан в топике |
| Участник | Чтение, отправка сообщений, загрузка файлов |

## Технологический стек (бюджетный)

### Вариант A: PWA + TWA (самый дешёвый)
- **Фронт:** React/Vue + PWA (Service Worker) — работает как приложение в браузере
- **Публикация в Store:** TWA (Trusted Web Activity) — обёртка PWA в APK для Google Play
- **Плюсы:** Один кодбаз, дешевле всего, обновления без ревью
- **Минусы:** Ограничения Push на iOS, меньше «нативности»

### Вариант B: React Native / Flutter
- **Фронт:** React Native (Expo) или Flutter
- **Плюсы:** Нативный вид, полноценные Push, публикация в App Store и Google Play
- **Минусы:** Дороже, дольше, нужна сборка под платформы

### Вариант C: Capacitor + веб
- **Фронт:** Vue/React SPA, обёртка в Capacitor
- **Плюсы:** Веб-код, один раз написал — везде, проще чем RN
- **Минусы:** Менее нативно, чем RN/Flutter

**Рекомендация при бюджете до 100к:** Вариант A (PWA) или C (Capacitor). Вариант B — если критичны App Store и нативный UX.

### Бэкенд (общий для всех вариантов)
- **Стек:** Go (Fiber/Gin) или Node.js (Fastify) + PostgreSQL + Redis
- **Реалтайм:** WebSocket (или Socket.io)
- **Файлы:** S3-совместимое хранилище (Selectel, VK Cloud) или локальный MinIO
- **Деплой:** VPS (Timeweb, Selectel, Reg.ru) или Railway/Render

## Вывод по бюджету «до 100к»

**Реалистично уложиться в 100 000 ₽, если:**
1. Ты делаешь фронт и/или бэк — сам.
2. Дизайн — простой, но аккуратный (фриланс или шаблон).
3. MVP без сложных фич (без видеозвонков, без сложной модерации).
4. PWA или Capacitor — без нативной разработки под iOS/Android.

**Не уложишься, если:**
- Всё отдаёшь на аутсорс.
- Нужны нативные приложения в App Store и Google Play с нуля.
- Сложные требования: энд-ту-энд шифрование, видеозвонки, тяжёлая модерация.

## Этапы разработки (MVP)

1. **Неделя 1–2:** Дизайн (экран входа, список топиков, экран чата).
2. **Неделя 2–4:** Бэкенд (auth, топики, сообщения, WebSocket).
3. **Неделя 4–6:** Фронт/PWA (страницы, реалтайм, загрузка файлов).
4. **Неделя 6–7:** Тестирование, деплой, публикация (TWA в Google Play).

## Риски и упрощения

| Риск | Упрощение |
|------|-----------|
| Модерация контента | Ручная модерация, жалобы вручную |
| Масштабирование | Начать с одного сервера, масштабировать при росте |
| Push на iOS | PWA на iOS ограничен — можно отложить или использовать Capacitor |
| AI-бот (@UroclubGPTbot) | Отдельный этап, после MVP |

## Следующие шаги

1. Выбрать стек: PWA vs Capacitor vs React Native.
2. Найти дизайнера или взять готовый UI-kit (например, для мессенджеров).
3. Спроектировать API (REST + WebSocket).
4. Начать с бэкенда или с прототипа фронта — по предпочтениям.

Если нужно, могу расписать конкретный API (эндпоинты, WebSocket-события) или структуру БД под этот сценарий.
